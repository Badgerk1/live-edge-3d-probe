# Contributing & Maintenance Guide

## Environment Restriction — "Direct GitHub API Calls Are Blocked"

Some AI-assisted development environments (including the GitHub Copilot coding
agent sandbox) cannot perform direct GitHub REST/GraphQL API calls such as
creating branches or refs via the API.  If you encounter the message:

> **Direct GitHub API calls are blocked in this environment**

the workaround is to use the **GitHub Actions workflow** included in this
repository.  Actions run inside GitHub's own infrastructure and are not subject
to the same network restrictions.

---

## Automated Version Bumps & Baseline Backups (GitHub Actions)

The workflow **`.github/workflows/version-bump.yml`** ("Version Bump & Baseline
Backup") handles both tasks from the GitHub UI — no local toolchain required.

### How to trigger

1. Open the repository on GitHub.
2. Go to the **Actions** tab.
3. Select **"Version Bump & Baseline Backup"** from the left sidebar.
4. Click **"Run workflow"** (top-right of the run list).
5. Fill in the inputs:

   | Input | Options | Default | Description |
   |-------|---------|---------|-------------|
   | `bump_type` | `patch` / `minor` / `major` | `patch` | Semver segment to increment |
   | `create_baseline` | `true` / `false` | `true` | Create a baseline backup branch after bumping |
   | `baseline_prefix` | any string | `baseline` | Prefix for the new backup branch name |

6. Click **"Run workflow"** to start.

The workflow will:

- Read the current `version` from `manifest.json`.
- Compute the next version according to `bump_type`.
- Update **all version strings** consistently across:
  - `manifest.json` — package version field
  - `src/js/core.js` — `SM_VERSION` global (drives the UI tag and exported metadata)
  - `src/config-body.html` — static fallback version strings in the HTML
  - `index.js` — plugin loaded/unloaded log messages
- Rebuild `config.html` via `./build.sh`.
- Commit and push directly to `main`.
- Optionally create a baseline branch named `<prefix>-<YYYY-MM-DD>-v<version>`.

### Version string locations

| File | Symbol / field | Example |
|------|---------------|---------|
| `manifest.json` | `"version"` | `"21.0.0"` |
| `src/js/core.js` | `SM_VERSION` | `'V21.0'` |
| `src/config-body.html` | `#sm-version-tag` span fallback | `V21.0` |
| `src/config-body.html` | mini footer label | `3D Live Edge Mesh V21.0` |
| `index.js` | startup log | `v21.0.0 plugin loaded` |
| `index.js` | shutdown log | `v21.0.0 plugin unloaded` |
| `src/js/probe-engine.js` | `version: '1.7.0'` | *(schema version — bump separately only when the mesh JSON schema changes)* |

> **Do not** edit `config.html` directly — it is fully regenerated by
> `./build.sh` from the files in `src/`.  Always edit the source files and
> rebuild.

## Manual Version Bump (no Actions / no network)

If you need to bump the version locally without triggering the workflow:

```bash
# 1. Edit the version strings (example: bumping to 21.1.0)
OLD=21.0.0  NEW=21.1.0
OLD_UI=V21.0  NEW_UI=V21.1

# manifest.json
sed -i "s/\"version\": \"$OLD\"/\"version\": \"$NEW\"/" manifest.json

# src/js/core.js
sed -i "s/var SM_VERSION = '$OLD_UI';/var SM_VERSION = '$NEW_UI';/" src/js/core.js

# src/config-body.html — two occurrences
sed -i "s/$OLD_UI/$NEW_UI/g" src/config-body.html

# index.js — startup + shutdown log lines
sed -i "s/v$OLD plugin/v$NEW plugin/g" index.js

# 2. Rebuild config.html
./build.sh

# 3. Commit and push
git add manifest.json src/js/core.js src/config-body.html index.js config.html
git commit -m "chore: bump version to $NEW"
git push
```

## Creating a Baseline Backup Branch Manually

```bash
git fetch origin
git checkout main && git pull
BRANCH="baseline-$(date -u '+%Y-%m-%d')-v$(jq -r .version manifest.json)"
git checkout -b "$BRANCH"
git push -u origin "$BRANCH"
```

## Security Notes

- The workflow runs with `contents: write` permission scoped to this repository only.
- It uses `actions/checkout@v4` (pinned major version) and no third-party action dependencies.
- All writes go to `main` and named baseline branches — no secrets are required or exposed.
- Branch protection rules on `main` still apply; if you have required status checks,
  add a bypass rule for `github-actions[bot]` or remove the rule for the bot actor.
